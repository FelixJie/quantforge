# 大模型（LLM）配置

## 默认 Provider：MiniMax-M3

全站 AI 能力（聊股对话、每日荐股、晨报、研报精读等）统一走 `ai_client.py` 的 provider 链，**默认且唯一主力是 MiniMax-M3**。

- 硬编码默认：`ai_client._DEFAULT_PRESET = "minimax-m3"`。
- 运行配置：`data/cache/llm_config.json` = `{"provider": "minimax-m3"}`（与硬编码一致；改硬编码是防 config 丢失时误用欠费的 DeepSeek）。
- 模型：`MiniMax-M3`，base_url `https://api.minimaxi.com/anthropic`（Anthropic 兼容协议）。

## 凭证来自 .env

MiniMax 的 key 用的是 **`ANTHROPIC_*`** 变量名（历史原因，base_url 已指向 minimaxi）：

```
ANTHROPIC_API_KEY=sk-cp-...        # MiniMax key（约 125 字符）
ANTHROPIC_BASE_URL=https://api.minimaxi.com/anthropic
```

`ai_client._PRESETS` 在 **import 时**就读 `os.getenv`，而仓库没装 python-dotenv、`run_backend.bat`/`启动.bat` 也不注入这些变量。因此 `app.py` 顶部用零依赖的 `_load_dotenv()`，**在任何路由 import 之前**把项目根 `.env` 灌进 `os.environ`（已存在的真实 env 不覆盖）。任意 `uvicorn` 起法都会自动读 .env。

> 症状提示：若 `/api/llm-stats/config` 里 minimax 的 key 掩码显示为 `****`（长度 ≤12 的空 key），说明 .env 没被加载 → MiniMax 报 `Authentication Fails (governor)` / `Could not resolve authentication method`。根因多半是后端**从非项目根 CWD 启动**或 .env 缺失。

## Fallback 链：跳过无 key 的 provider

`_fallback_chain()` 以 effective provider（默认 MiniMax）打头，再接其它预设兜底。**但会跳过 `api_key` 为空的预设**：

- DeepSeek 账号长期欠费（402）且多半没配 `DEEPSEEK_API_KEY` → 空 key → **不进链**，不再产生满屏 `deepseek Authentication Fails` 噪声日志。
- 一旦配上 `DEEPSEEK_API_KEY`，它会自动重新作为兜底加入链尾。

即：当前实际就是 **MiniMax 单一 provider**。

## 临时过载重试

MiniMax 高峰会轮流抛 529/500/502/503/429 等瞬时错。`_is_overloaded()` 判定后对**同一 provider** 退避重试（`_OVERLOAD_RETRIES=3`，backoff `[2,5,10]s`），**不含** 402/401/400 这类硬错。避免一遇过载就跌到欠费的备用号。

## 如何切换 provider

- 临时（不改代码）：改 `data/cache/llm_config.json` 的 `provider` 字段，或在 `/api/llm-stats/config` PUT 配置。
- 永久默认：改 `ai_client._DEFAULT_PRESET`。
- 新增 provider：在 `ai_client._PRESETS` 加一项（label/base_url/api_key/model），配好对应环境变量即可。
